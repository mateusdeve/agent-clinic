    **Migration 003 — `003_enable_rls_policies.py`**
    `revision = "003"`, `down_revision = "002"`

    Per D-10 (PostgreSQL RLS), research Pattern 6:

    upgrade():
    ```sql
    -- Helper function to read tenant context from session variable
    CREATE OR REPLACE FUNCTION current_tenant_id()
    RETURNS UUID AS $$
    BEGIN
        RETURN NULLIF(current_setting('app.tenant_id', TRUE), '')::UUID;
    END;
    $$ LANGUAGE plpgsql STABLE SECURITY DEFINER;

    -- Enable RLS and create isolation policies on ALL data tables
    -- Tables: patients, appointments, doctors, conversations,
    --         conversation_summaries, knowledge_chunks, follow_ups, users
    ```

    For EACH table in [patients, appointments, doctors, conversations, conversation_summaries, knowledge_chunks, follow_ups, users]:
    ```sql
    ALTER TABLE {table} ENABLE ROW LEVEL SECURITY;
    CREATE POLICY tenant_isolation_{table} ON {table}
        USING (tenant_id = current_tenant_id())
        WITH CHECK (tenant_id = current_tenant_id());
    ```

    Per research anti-pattern guidance, the WhatsApp bot DB user needs BYPASSRLS:
    ```sql
    -- Grant BYPASSRLS to the application DB user so the WhatsApp bot
    -- (which has no tenant context) continues to work unchanged (D-11)
    -- The web API will SET LOCAL app.tenant_id before queries
    DO $$
    BEGIN
        -- Get current DB user and grant BYPASSRLS
        EXECUTE format('ALTER USER %I WITH BYPASSRLS', current_user);
    END $$;
    ```

    Note: This grants BYPASSRLS to the current DB user. In production with separate users, the web API user should NOT have BYPASSRLS — only the bot user. For v1 single-user setup, this is acceptable.

    downgrade():
    ```sql

    ```
